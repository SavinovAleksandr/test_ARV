# План улучшений работы программы «Модуль оценки АРВ»

Ниже перечислены возможные улучшения по категориям. **Код пока не меняется** — план предназначен для согласования.

---

## 1. Надёжность и валидация входных данных

| № | Проблема | Предложение |
|---|----------|-------------|
| 1.1 | При нажатии «Расчёт» без предварительного Drag&Drop обращение к `fInf.files` / `fInf.xlxs` может привести к `NullReferenceException` или неочевидному поведению. | Перед началом расчёта проверять: `fInf.files` не null/пустой; в наборе есть .xlsx с листами rgms, rems, ut, arv, grf; для каждого режима из xlsx задан соответствующий .rst. При невыполнении — сообщение пользователю («Сначала перетащите файлы…» / «В файле задания не найдены режимы…») и не запускать расчёт. |
| 1.2 | Не проверяется существование файлов на диске (fInf.scns, пути из xlxs, .sch, .ut2). При перемещении/удалении файла после Drop — исключение в процессе расчёта. | Перед циклом расчёта проверять существование всех используемых файлов (scn, rst, sch, ut2, xlsx). При отсутствии — сообщить, какой файл не найден, и не запускать расчёт. |
| 1.3 | В `files_drop` при отсутствии в наборе .scn или .xlsx обращение к `fInf.scns` / `fInf.xlxs.name` может дать null или пустую строку. | В геттерах `compModel` (scns, xlxs) и при подстановке в UI предусмотреть защиту от null; в Drop не выводить имя, если файл не найден, или показывать «—». |

## 2. Отзывчивость интерфейса (UI не «зависает»)

| № | Проблема | Предложение |
|---|----------|-------------|
| 2.1 | Весь расчёт выполняется в обработчике кнопки «Расчёт» (в потоке UI). При большом числе режимов окно не реагирует, «Не отвечает». | Вынести расчётный цикл в фоновый поток (например, `Task.Run` или `BackgroundWorker`). Сводку по завершении и обновление прогресс-бара выполнять через `Dispatcher.Invoke` (уже частично есть для prBar). |
| 2.2 | Нет возможности прервать длительный расчёт. | Добавить кнопку «Остановить» (активна во время расчёта). Флаг отмены проверять в цикле по режимам; при установке — корректно завершить цикл, сохранить уже сформированные xlsx/docx и сообщить «Расчёт прерван пользователем». |

## 3. Обработка ошибок и диагностика

| № | Проблема | Предложение |
|---|----------|-------------|
| 3.1 | В `catch` при ошибке расчёта пользователю показывается только общий текст («Ошибка работы алгоритма Модуля») без причины. Исключение не логируется. | В сообщении об ошибке выводить краткое описание (ex.Message). Дополнительно: запись полного исключения (и при необходимости stack trace) в файл лога в каталоге программы или в отдельную папку. |
| 3.2 | При любой ошибке в блоке расчёта предлагается завершить приложение (`Application.Current.Shutdown()`). | Не завершать приложение по умолчанию: показать ошибку, разблокировать кнопку «Расчёт» и курсор. Опцию «Закрыть программу» оставить только при явном выборе пользователя (например, вторая кнопка в диалоге). |
| 3.3 | COM-объекты (Rastr, Word): при исключении в середине цикла процесс WinWord и/или Rastr может остаться в памяти. | В блоках try/finally (или using + обёртка) гарантировать вызов Quit/освобождение COM для Word и Rastr при любом выходе (в т.ч. при отмене или ошибке). |

## 4. Производительность

| № | Проблема | Предложение |
|---|----------|-------------|
| 4.1 | Стили Excel задаются на весь диапазон `A1:XFD1048576` (все ячейки листа), что очень тяжело. | Применять стили только к используемому диапазону (например, до последней заполненной строки и 6 столбцов) или к фиксированному разумному диапазону (например, A1:F10000). |
| 4.2 | Файл задания xlsx при каждом обращении к `fInf.xlxs` заново читается и парсится с диска. | Кэшировать результат разбора xlsx (например, при первом обращении после установки `files`), инвалидировать кэш при смене набора файлов (новый Drop). |
| 4.3 | В предварительном цикле (до диалога «Начать расчет?») для каждого режима создаётся отдельный экземпляр Rastr, настраивается и сохраняется — при большом числе режимов задержка до первого диалога. | Рассмотреть: один экземпляр Rastr с переиспользованием в предварительном цикле (если API позволяет) или перенос части настройки в основной цикл; либо вынести предварительный этап в фон с индикацией «Подготовка…». |

## 5. Удобство использования (UX)

| № | Проблема | Предложение |
|---|----------|-------------|
| 5.1 | Диалог «Начать расчет?» появляется только после тяжёлой предварительной настройки всех режимов; при отказе пользователь уже ждал зря. | Сначала показывать диалог подтверждения, затем выполнять предварительную настройку и основной расчёт (при отказе — только разблокировать кнопку и курсор). |
| 5.2 | Нет явной индикации текущего этапа (какой режим/ремонтная схема считаются). | Рядом с прогресс-баром выводить текст вида «Режим X из Y: &lt;имя режима&gt;» (обновлять через Dispatcher из фонового потока). |
| 5.3 | Пути к шаблонам Rastr зашиты в код (Documents\RastrWin3\SHABLON\…). При другой установке Rastr или другом языке системы путь может быть неверным. | Вынести базовый каталог шаблонов в настройки (файл конфигурации или диалог «Настройки»), с подстановкой по умолчанию текущего значения. |
| 5.4 | Результаты всегда сохраняются в каталог программы с фиксированными именами; при повторном расчёте перезаписываются без предупреждения. | Опция «Сохранять в указанную папку» + выбор папки (и при желании шаблон имени); либо предупреждение при перезаписи существующих «Приложение. Результат…». |

## 6. Поддерживаемость кода

| № | Проблема | Предложение |
|---|----------|-------------|
| 6.1 | Очень длинный метод `start_Click` (свыше 300 строк), смешаны настройка Rastr, циклы, Excel, Word, прогресс, обработка ошибок. | Разбить на отдельные методы по зонам ответственности: проверка входных данных, предварительная настройка режимов, цикл расчёта (с вызовом get_pss_quality/get_mpd), формирование и сохранение отчётов, показ сообщений. Это упростит тестирование и изменение логики. |
| 6.2 | Магические числа и строки (пути, имена листов, ключи таблиц Rastr) разбросаны по коду. | Вынести в константы или конфиг: имена листов xlsx, имена таблиц/колонок Rastr, пути шаблонов, форматы имён выходных файлов. |
| 6.3 | Дублирование: два цикла по `fInf.xlxs.rgms.Where(...)` — один для предварительной настройки, другой для расчёта; повторяющиеся вызовы FirstOrDefault по ut/sch. | Вынести общий перечисляемый список режимов в переменную; для ut/sch по id — вспомогательные методы или локальные переменные, чтобы не вызывать Where().FirstOrDefault() многократно в одном контексте. |

## 7. Безопасность и окружение

| № | Проблема | Предложение |
|---|----------|-------------|
| 7.1 | Запись DLL из ресурсов в каталог программы при первом запуске может быть запрещена политикой или правами. | Уже есть сообщение об ошибке; дополнительно: проверять права на запись в каталог до начала расчёта (например, попытка создать тестовый файл и удалить) и предупреждать пользователя заранее. |
| 7.2 | Зависимость от установленного Word (COM). На машине без Word приложение упадёт при создании графического отчёта. | Проверять наличие Word при старте или перед первым использованием (например, попытка получить TypeFromCLSID); при отсутствии — сообщение «Требуется Microsoft Word для формирования графического отчёта» и опция продолжить только табличный отчёт (xlsx). |

## Приоритизация (для согласования)

Предлагаемая очередность с учётом эффекта и трудозатрат:

1. **Высокий приоритет (стабильность и предсказуемость):**  
   1.1, 1.2 — валидация перед расчётом; 3.1, 3.2 — обработка ошибок и логирование; 3.3 — освобождение COM.

2. **Средний приоритет (удобство и производительность):**  
   2.1 — фоновый расчёт; 4.1 — стили Excel по диапазону; 5.1 — диалог до тяжёлой подготовки; 5.2 — индикация текущего этапа.

3. **Ниже (по возможности):**  
   2.2 — отмена расчёта; 1.3, 4.2, 4.3 — кэш xlsx и оптимизация Rastr; 5.3, 5.4 — настройки путей и папки результатов; 6.1–6.3 — рефакторинг; 7.1, 7.2 — проверки окружения.
